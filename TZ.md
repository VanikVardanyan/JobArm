# Техническое задание (MVP)

## 1. Название проекта

`jobNow` — веб-сервис для быстрого размещения и поиска разовых задач и подработок по всей Армении.

Примеры задач:
- нужен грузчик на 1-2 часа
- помочь поднять диван на 2 этаж
- нужен человек на разгрузку
- мелкая помощь по дому
- срочная подработка рядом

## 2. Идея продукта

Заказчик заходит в сервис, быстро авторизуется, создает карточку задачи и публикует ее.
Любой другой пользователь без регистрации может открыть список объявлений, посмотреть детали и позвонить заказчику.

Главная ценность сервиса:
- очень быстрое размещение заявки
- минимальный порог входа
- локальные задачи "здесь и сейчас"
- открытый доступ к просмотру заявок
- простая связь между сторонами через звонок

## 3. Цель MVP

Проверить, нужен ли в Армении простой сервис для разовых бытовых задач и быстрых подработок без сложной регистрации, чатов и биржевой логики.

MVP должен ответить на вопросы:
- готовы ли люди быстро публиковать такие заявки
- будут ли другие пользователи регулярно звонить по объявлениям
- какие категории задач самые востребованные
- достаточно ли звонка как основного способа связи
- нужен ли вообще отдельный аккаунт исполнителя

## 4. География запуска

Первый рынок: вся Армения.

Требования к географии:
- при создании заявки заказчик обязан выбрать `регион`
- заказчик обязан указать `адрес`
- по умолчанию в форме выбран `Ереван`

## 5. Формат продукта

На старте продукт реализуется как:
- мобильное веб-приложение

Это даст:
- быстрый запуск
- доступ без установки
- простую публикацию с телефона
- низкую стоимость первой версии

Позже можно развивать:
- iOS-приложение
- Android-приложение

## 6. Модель пользователей MVP

В MVP есть только один тип пользователя с аккаунтом:
- `заказчик` — автор заявки

Также есть второй тип использования без аккаунта:
- `гость` — любой пользователь, который просматривает заявки и звонит

Важно:
- отдельной роли `исполнитель` в системе MVP нет
- регистрация для просмотра заявок не требуется
- регистрация нужна только для создания и управления своими объявлениями

## 7. Авторизация и профиль

### 7.1. Авторизация MVP

Для MVP используется:
- вход через `Google Sign-In`

Авторизация нужна только для действий:
- создать заявку
- редактировать свою заявку
- закрыть свою заявку
- смотреть список своих заявок

### 7.2. Телефон в MVP

При первой публикации заявки заказчик обязан указать:
- имя
- номер телефона

Правила MVP:
- номер телефона сохраняется в профиле заказчика
- номер телефона используется как основной контакт в заявке
- номер телефона в MVP не подтверждается по SMS
- SMS-авторизация и SMS-верификация откладываются на будущую версию

Причина такого решения:
- быстрее запуск
- ниже стоимость MVP
- меньше трения при входе

## 8. Основной пользовательский сценарий

### Сценарий заказчика
1. Пользователь открывает сайт.
2. Нажимает `Создать заявку`.
3. Если не авторизован, входит через Google.
4. При первой публикации указывает имя и номер телефона.
5. Заполняет короткую форму заявки.
6. Выбирает регион.
7. Указывает адрес.
8. Публикует заявку.
9. Получает звонки от заинтересованных пользователей.
10. После выполнения закрывает заявку.

### Сценарий гостя
1. Пользователь открывает сайт.
2. Смотрит список активных заявок.
3. Использует фильтры.
4. Открывает карточку интересной заявки.
5. Видит описание, цену, регион, адрес, время и телефон.
6. Нажимает `Позвонить`.

## 9. Функциональные требования MVP

### 9.1. Авторизация
- вход через Google
- выход из аккаунта
- сохранение профиля заказчика

### 9.2. Профиль заказчика
- имя
- email из Google
- номер телефона
- avatar URL из Google опционально
- createdAt

### 9.3. Создание заявки

Поля карточки:
- заголовок
- описание
- категория
- цена
- тип цены: фиксированная / договорная / почасовая
- регион
- адрес
- дата
- время
- срочно / не срочно
- номер телефона для связи

Требования к форме:
- `регион` обязателен
- `адрес` обязателен
- по умолчанию выбран `Ереван`
- форма должна быть короткой и понятной
- создание заявки должно занимать не более 1-2 минут

### 9.4. Категории задач

Стартовые категории:
- грузчики
- помощь при переезде
- разгрузка / погрузка
- бытовая помощь
- уборка
- мелкий ремонт
- доставка
- другое

### 9.5. Лента заявок
- список свежих активных заявок
- сортировка по новизне
- карточки с краткой информацией
- отображение цены
- отображение региона
- отображение времени публикации
- отметка `срочно`
- доступна без авторизации

### 9.6. Поиск и фильтры
- поиск по ключевым словам
- фильтр по категории
- фильтр по региону
- фильтр по цене
- фильтр по срочности
- фильтр только по активным заявкам

### 9.7. Страница заявки
- полный заголовок
- полное описание
- цена
- регион
- адрес
- дата и время
- имя заказчика
- номер телефона
- кнопка `Позвонить`
- статус заявки

### 9.8. Управление своими заявками
Заказчик может:
- редактировать свою заявку
- закрыть заявку
- удалить заявку
- отметить заявку как выполненную
- смотреть список своих заявок

### 9.9. Админ-функции
Администратор может:
- смотреть всех заказчиков
- смотреть все заявки
- скрывать сомнительные заявки
- блокировать пользователей
- редактировать категории

## 10. Нефункциональные требования

### 10.1. Скорость
- сайт должен быстро открываться на мобильном интернете
- список заявок должен загружаться менее чем за 2 секунды при базовой нагрузке

### 10.2. Простота
- ключевой сценарий должен быть понятен без обучения
- публикация заявки должна быть максимально короткой
- просмотр заявок должен быть доступен сразу, без регистрации

### 10.3. Адаптивность
- приоритет мобильной версии
- корректная работа на смартфонах
- базовая поддержка десктопа

### 10.4. Языки
На старте желательно предусмотреть:
- русский
- армянский

Позже можно добавить:
- английский

### 10.5. Безопасность
- базовая валидация телефона
- защита от спама
- логирование действий
- ограничение частоты публикаций
- возможность ручной модерации

## 11. Ограничения MVP

Чтобы быстрее проверить гипотезу, в первую версию не включать:
- регистрацию исполнителей
- встроенный чат
- онлайн-оплату
- сложную систему рейтингов
- безопасную сделку
- геолокацию в реальном времени
- SMS-верификацию телефона
- автоматическое назначение исполнителя

Связь в MVP:
- только через звонок по номеру телефона

## 12. Метрики успеха MVP

Важно отслеживать:
- количество новых заявок в день
- количество входов заказчиков через Google
- количество просмотров карточек
- количество нажатий на кнопку звонка
- долю закрытых заявок
- популярные категории
- популярные регионы
- среднее время жизни заявки

## 13. Возможные риски

- фейковые или спам-заявки
- пользователи могут указывать неверный номер телефона
- часть пользователей может ожидать чат внутри приложения
- может быть мало заявок на старте без локального продвижения
- может понадобиться ручная модерация уже в начале
- часть пользователей может не захотеть входить через Google

## 17. Roadmap разработки

### Этап 1. Подготовка
- утвердить название
- утвердить языки MVP
- утвердить список категорий
- утвердить список регионов Армении
- утвердить финальный набор полей заявки

### Этап 2. MVP разработка
- Google-авторизация заказчика
- создание и просмотр заявок
- поиск и фильтры
- страница заявки
- страница `Мои заявки`
- закрытие и редактирование заявки
- базовая админка или ручная модерация

### Этап 3. Тестовый запуск
- запуск на всю Армению
- ручная модерация
- сбор обратной связи
- анализ самых популярных сценариев
- анализ регионов и категорий

### Этап 4. Улучшения после MVP
- SMS-верификация телефона
- чат
- отзывы
- рейтинг
- продвижение заявок
- отдельный сценарий для исполнителей при необходимости

## 18. Что важно решить до начала разработки

1. Показывать ли номер телефона сразу в карточке или только после открытия заявки.
2. Нужен ли точный адрес или достаточно адреса в свободной форме.
3. Нужна ли премодерация каждой новой заявки.
4. Какие категории запускать в первой версии.
5. Нужен ли в будущем вход не только через Google, но и через Apple или телефон.
6. Какая модель монетизации планируется позже:
- платное поднятие заявки
- платное размещение для бизнеса
- подписка
- комиссия

## 19. Рекомендуемый фокус первой версии

Лучше не распыляться и сделать первую версию вокруг одного главного кейса:

"Нужен человек на короткую задачу сегодня или сейчас".

Это поможет:
- проще объяснить продукт
- быстрее проверить спрос
- не усложнять интерфейс
- быстрее выйти на рынок

## 20. Итог

`jobNow` в MVP — это простой сервис быстрых локальных задач по всей Армении, где автор заявки входит через Google, быстро публикует объявление с телефоном, регионом и адресом, а любой пользователь без регистрации может открыть заявку и позвонить.

Главный принцип первой версии:
- минимум действий
- максимум скорости
- открытый просмотр без регистрации
- мобильный фокус
- запуск на всю Армению
